# Contributing to delegate-flow

## The one rule that prevents drift

`flows/lib/schemas.js` is the **single source of truth** for output contracts.
Everything else is generated or mirrors it:

- `schemas/*.json` ← generated by `node tools/gen-schemas.mjs`
- `dist/*.flow.js` ← bundled by `node tools/build-flows.mjs`

Never hand-edit `schemas/*.json` or `dist/*.flow.js`. Edit the `.js` source and rebuild.

## Keeping in step with delegate-kit

delegate-flow implements the protocol delegate-kit defines. If a delegate-kit brief
field or the universal envelope changes:

1. Mirror it in `flows/lib/schemas.js` (envelope/role fields) and
   `flows/lib/briefs.js` (brief builders).
2. `npm run build && npm run check`.
3. Note the sync in `.agent-memory/DECISIONS.md`.

## Adding a flow

1. Create `flows/<name>.flow.js`. First line must be `export const meta = {...}`
   (pure literal, titles matching `phases`).
2. Add a `// @uses lib/a.js lib/b.js` directive listing the libs to inline.
3. `import` those libs normally — the bundler strips the imports and inlines the code.
4. Use only the documented Workflow hooks (`agent`, `parallel`, `pipeline`, `phase`,
   `log`, `args`, `budget`). No `Date.now()`/`Math.random()`/`new Date()` — they break
   resume and the engine blocks them.
5. `npm run build && npm run check`, then run it through the `Workflow` tool with a
   small `args` to smoke-test.

## Cross-platform discipline (inherited from delegate-kit)

- Briefs must be self-contained: no "as discussed", no parent-chat references.
- No assuming `cd` persists; use repo-relative or absolute paths.
- Avoid `sed -i`, `xargs -0`, `osascript`, PowerShell-only one-liners in any command a
  subagent or `tools/` script runs.

## Tests

`tools/check.sh` is the gate: it regenerates schemas, bundles flows, asserts each dist
starts with `meta`, asserts no leftover imports, and scans briefs for `cd .`. Run it
before every commit.

> Do **not** validate bundled flows with `node --check`. The Workflow engine wraps each
> script in an async function, so top-level `return`/`await` are legal there but illegal to
> a bare module parser — `node --check` will false-fail. The real validator is running the
> flow through the `Workflow` tool with small `args`.
